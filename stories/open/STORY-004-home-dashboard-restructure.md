## Background

当前首页已经接入了较多信息，包括：
- 用户状态
- 最近题目
- 最近提交
- 最近一次失败原因
- 最近一次通过记录
- 比赛快照
- 一些统计摘要

但首页目前的问题是“信息太多，但缺少稳定的信息层次”，具体表现为：
- 首页像多个模块的临时拼接，缺少明确主线
- 页面区块较多，但用户进入后不容易快速判断“下一步应该做什么”
- 对普通用户、教师、管理员而言，首页提供的信息没有明显分层
- 部分模块更像调试看板或信息板，而不是清晰的门户页
- 信息密度不均衡，有些区域信息很多，有些区域价值较低
- 卡片样式虽已统一一部分，但首页整体仍缺少明确视觉重心

这个问题不会影响系统功能是否可用，但会直接影响：
- 首次进入系统的印象
- 课堂展示效果
- 不同角色使用系统时的效率

## Goal

将首页重构为一个真正的“门户页 / dashboard”，明确页面主线和信息优先级。

首页应回答这些问题：
- 当前用户最该做什么？
- 最近有哪些重要动态？
- 对应角色最常用的入口在哪里？
- 当前系统状态是否一目了然？

最终目标不是“堆更多卡片”，而是让首页形成稳定结构：
- 顶部是身份与当前重点
- 中间是核心工作流入口
- 底部是补充信息和次级模块

## Scope

本次包含：

### 1. 首页信息架构重排
重新定义首页各区块优先级，建议按以下层次组织：
- 顶部：欢迎区 / 当前身份 / 关键入口
- 中部：当前工作流（最近提交、继续做题、最近比赛、最近作业）
- 底部：公告、论坛动态、次级统计

要求：
- 明确主区块和次区块
- 避免所有信息平铺并列
- 保证用户进入首页后能快速找到下一步动作

### 2. 普通用户视角优化
针对普通用户，首页应重点突出：
- 继续做题
- 最近提交结果
- 最近参与的比赛或作业
- 推荐进入题库 / 比赛 / 题单的入口

需要弱化：
- 对普通用户不直接产生价值的管理型统计信息

### 3. 教师 / 管理员视角优化
针对教师和管理员，首页应提供更明确的快捷入口，例如：
- 教师：班级、作业、题单
- 管理员：题目、比赛、提交、公告管理

要求：
- 入口清晰，不应埋在过深的卡片区块里
- 不同角色进入首页时，首页重点不同

### 4. 继续工作流优化
首页应保留并强化当前已有的“继续上次工作”能力，但要更清晰。

建议保留并重构这些能力：
- 最近一次提交
- 最近一次失败原因
- 最近一次 Accepted
- Continue Last Work

要求：
- 同一类信息不要重复出现多块
- 同一动作的入口不要分散在多个位置

### 5. 信息去冗余
需要审查当前首页中的低价值或重复信息，并决定：
- 合并
- 降级展示
- 移除

重点关注：
- 是否存在重复表达同一状态的卡片
- 是否存在“看起来很多，但用户不关心”的统计块
- 是否存在只对开发调试有价值、对展示和使用价值不高的内容

### 6. 首页视觉重心优化
首页需要形成更稳定的视觉节奏：
- 主卡片更突出
- 次级卡片弱化
- 区块标题更清晰
- 卡片高度和排列更规整

目标不是大改主题色，而是让首页从“拼接信息板”更接近“系统门户”。

### 7. 首页模块边界清理
明确哪些信息应该留在首页，哪些更适合留在二级页，例如：
- 详细统计更适合放在 profile / admin 页面
- 详细榜单更适合放在 rankings 页面
- 详细比赛内容更适合放在 contests 页面

首页应保留的是入口和摘要，而不是把详细页内容搬过来。

## Out of Scope

本次不包含：
- 新增新的后端统计接口
- 大规模重做导航栏
- 重做题库页、比赛页、教学页
- 新增复杂推荐系统
- 引入新的角色体系

## Specific Improvement Directions

建议至少从以下方向中选择并落地：

### 1. 顶部欢迎区重构
- 当前用户身份更清晰
- 显示用户名、角色、关键入口
- 将“继续工作”的主按钮放在更稳定的位置

### 2. 首页模块重新分组
建议分为：
- Quick Actions
- Continue Work
- Recent Activity
- Contest Snapshot / Teaching Snapshot
- Announcements

### 3. 角色化首页入口
- 普通用户：首页强调题库、比赛、提交
- 教师：首页强调班级、作业、题单
- 管理员：首页强调题目、比赛、提交、公告管理

### 4. 最近活动区重构
- 最近提交
- 最近一次失败
- 最近一次 Accepted
不应完全分散，应形成一个统一的 activity 区块

### 5. 次级统计降权
将次级统计信息从主视觉区移开，避免与主工作流争夺注意力。

### 6. 公告和论坛摘要更轻量
首页保留摘要即可，不应变成公告页或论坛页的替代。

### 7. 布局规整化
- 卡片高度尽量统一
- 区块留白统一
- 不同类型信息不要采用完全相同的视觉权重

## Expected Behavior

完成后，首页至少应满足：
- 用户进入后能快速知道下一步可做什么
- 不同角色能看到更贴合自身任务的快捷入口
- 首页不再显得像多个模块的临时堆叠
- 最近活动信息集中而不重复
- 公告、比赛、教学、论坛等信息有明确主次关系

## Acceptance Criteria

- [ ] 首页信息层次比当前更清晰
- [ ] 普通用户、教师、管理员的首页重点有所区分
- [ ] “继续工作”相关信息不再重复分散
- [ ] 首页保留摘要与入口，不承担过多详细页职责
- [ ] 首页视觉结构更规整，主次关系明确
- [ ] 前端脚本语法检查通过

## Suggested Frontend Tech

本 story 允许使用轻量前端渲染技术增强首页展示，但不建议为了首页重构整体前端架构。

推荐技术：
- `dayjs`：统一首页中比赛、提交、公告、作业等时间的展示
- `ECharts` 或 `Chart.js`：如需将次级统计改成轻量图表，可在首页局部接入

技术约束：
- 图表只作为摘要增强，不替代详细统计页
- 不引入整站级 UI 框架
- 不为首页单独重写新的渲染体系

## Risks / Notes

- 首页重构容易牵涉多个已有模块，必须控制范围
- 不要在本 story 中同时修改过多二级页逻辑
- 如果需要删减首页某些区块，应在 delivery 中说明删减理由

## Handover Notes

这是一个纯前端展示 story，重点在信息架构和视觉层次，不要求新增后端能力。

如果时间有限，优先级建议如下：
1. 首页模块重排
2. Continue Work 区块重构
3. 角色化快捷入口
4. 次级统计降权
5. 公告 / 论坛摘要轻量化
