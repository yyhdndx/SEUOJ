# SEU OJ 项目答辩问答整理

本文档用于支持 SEU OJ 项目立项/中期/结题答辩中的问答环节整理，内容基于当前仓库中已经存在的代码、接口、需求建模文档和前后端结构，而不是脱离实现的空泛描述。

建议把这份文档作为答辩准备底稿使用：

- 汇报时不需要逐字照念
- 可以优先记住每题的“回答思路”
- 再根据现场时间决定是否补充“示例回答”和“追问应对”

---

## 1. 项目现状总述

在正式回答老师提问前，建议先统一项目定位。当前项目更适合被定义为：

> SEU OJ 是一个面向训练、比赛、教学与讨论场景的综合平台，不只是单一的在线判题网站，而是“在线评测平台 + 比赛系统 + 教学管理系统 + 社区交流系统”的组合。

从当前仓库可见，系统已经覆盖以下核心能力：

- 用户注册、登录、身份与权限控制
- 题库浏览、题目详情、题解展示
- 样例运行、代码提交、异步判题、提交结果查看
- 比赛报名、比赛题目、比赛公告、比赛榜单
- 题单、班级、作业/考试、教师管理能力
- 公告、论坛、回帖、内容管理
- 管理端用户/题目/比赛/公告/提交治理

当前技术实现已经具备原型系统特征：

- 后端：Go + Gin + Gorm
- 数据层：MySQL
- 队列：Redis
- 判题隔离：Docker 沙箱
- 前端：静态页面 + 按业务域拆分的 JS/CSS

截至当前代码状态，后端已经可以通过 `go build ./...` 编译，这说明项目的技术路线已经具备可行性验证基础。

## 2. 答辩总原则

老师问这类问题时，通常不是只想听“有没有做”，而是想判断以下几点：

- 你们是否真正理解项目，而不是只会堆功能
- 需求、设计、实现之间是否能对上
- 团队是否具备继续推进和交付的能力
- 你们是否意识到风险，并有控制方案

因此回答时建议统一采用以下表达方式：

1. 先给结论
2. 再给依据
3. 最后补充当前边界或后续计划

尽量避免：

- 只说“我们觉得可以”
- 只列技术名词，不解释为什么这样设计
- 把未完成部分说得像已经完全实现

### Q1. 你们的团队是否具备管理/技术专长？如何处理人员变动（生病、缺席等）？

#### 老师想听什么

老师主要想确认：

- 团队是否有明确分工
- 项目是否依赖某一个人单点推进
- 出现成员缺席时是否还能继续

#### 回答思路

这一题不要只说“我们都会编程”。更好的答法是强调：

- 已经进行了按模块划分的协作
- 文档和接口足够支持交接
- 前后端结构本身就是为了多人协作而设计

#### 推荐回答

我们团队具备基本的前端、后端、数据库和系统联调能力，而且已经按业务域进行了模块化分工。比如题库与判题链路、比赛模块、教学模块、论坛与管理端可以相对独立推进，前端代码也已经从单一大文件拆分为按业务域组织的 JS 和 CSS，目的是降低多人协作时的冲突成本。

在人员变动方面，我们不是靠“某个人记得所有细节”来推进，而是靠接口文档、需求建模文档和模块边界做协作。即使某位成员因为生病或缺席暂时无法继续，其他成员也可以根据已有文档和代码结构接手对应模块，因此项目不会因为单点人员问题而整体停滞。

#### 可以补充的一句话

> 我们在组织方式上尽量让项目“可交接、可替补、可并行”，而不是依赖个人记忆推进。

#### 如果老师追问“你们具体怎么分工”

可以回答：

- 后端同学重点负责认证、题目、提交判题、比赛、教学等 API
- 前端同学按题库/比赛/教学/论坛/管理端拆分页面逻辑
- 公共部分由一位成员统一维护，例如路由入口、全局状态、基础样式和文档

### Q2. 项目的好处是什么？

#### 老师想听什么

这一题本质上是问“为什么值得做”，不是问“你们做了多少页面”。

#### 回答思路

最好从不同角色的收益来讲：

- 学生得到什么
- 教师得到什么
- 管理员得到什么

最后再落到平台整体价值。

#### 推荐回答

这个项目的核心好处是把原本分散的训练、比赛、作业和讨论过程整合到了一个统一平台里。对学生来说，可以在同一个系统里完成做题、提交、查看结果、参加比赛、加入班级、完成作业和参与讨论；对教师来说，可以创建题单、管理班级、布置作业或考试，并跟踪学生完成情况；对管理员来说，可以统一管理题目、比赛、公告、论坛和用户。

从平台层面看，这个项目的价值不是“再做一个刷题网站”，而是构建一个面向课程教学、竞赛训练和日常练习的完整闭环系统。

#### 可以补充的一句话

> 这个系统降低了学生和教师在多个平台之间切换的成本，也提升了训练和教学数据的统一管理能力。

### Q3. 项目技术上可行吗？项目的范围是什么？

#### 老师想听什么

老师要判断两件事：

- 技术路线是不是落地的
- 你们有没有把项目边界控制住

#### 回答思路

这一题一定要先讲“为什么可行”，而且要讲出已经有的实现证据。然后明确说明“本阶段做什么，不做什么”。

#### 推荐回答

从当前实现情况来看，项目技术上是可行的。我们已经完成了 Go 后端、MySQL 数据持久化、Redis 判题队列、Docker 沙箱判题、前端页面与后端 API 的对接，题库浏览、代码提交、异步判题、比赛、教学和论坛等主流程都已经在代码层面落地，而不是停留在设计阶段。后端当前也已经可以完成编译，这说明核心架构是连通的。

项目范围方面，我们本阶段聚焦的是“训练 - 判题 - 比赛 - 教学 - 讨论”这条主业务链路，具体包括用户与权限、题库与题解、提交与判题、比赛与榜单、题单/班级/作业、公告与论坛、管理后台。当前不把支付、社交私信、第三方统一登录、移动端 App、超大规模分布式调度等能力纳入本阶段核心范围。

#### 推荐强调点

> 我们不是追求一次性覆盖所有商业 OJ 的高级能力，而是优先把课程项目最核心的业务闭环做完整。

#### 如果老师追问“为什么说可行”

可以补充：

- 路由、权限中间件、服务层和数据库表已经形成对应关系
- 提交后写入数据库并进入 Redis 队列
- 独立 judge worker 从队列中消费任务并通过 Docker 沙箱执行代码
- 结果会回写数据库并由前端轮询展示

### Q4. 需要哪些资源（人力、设备、工具、知识产权）和预算？

#### 老师想听什么

老师不是只想听“要电脑”，而是看你们是否知道项目的真实成本构成。

#### 回答思路

按五类回答最清楚：

- 人力
- 设备
- 工具
- 知识产权
- 预算

#### 推荐回答

从人力上看，项目需要前端开发、后端开发、数据库设计、系统联调和文档整理几类工作，虽然是课程项目，但本质上已经覆盖了一个完整 Web 系统的基本分工。

从设备上看，需要能够运行开发环境的电脑，以及支持 MySQL、Redis 和 Docker 的本地环境或服务器环境。由于当前项目主要采用本地开发和本地运行原型，硬件门槛并不高，普通开发电脑即可支持。

从工具上看，主要使用 Go、Gin、Gorm、MySQL、Redis、Docker、Git 等开发和运行工具，这些工具大多是开源和免费可用的，因此软件工具成本较低。

从知识产权角度看，项目的业务代码、数据库结构、前端页面和需求文档由团队自主整理和实现；第三方依赖主要是开源框架和基础镜像，使用时按照对应许可证规范使用。

从预算上看，如果仅作为课程项目原型，本地开发和演示的预算可以控制在较低水平，主要成本来自时间投入而不是软件采购。如果后续需要公网部署，再根据服务器配置增加少量云资源成本即可。

#### 简化版回答

> 这个项目是“人力成本高于工具成本”的类型，技术栈以开源组件为主，因此预算压力主要不在购买工具，而在开发和联调时间。

### Q5. 有哪些风险？如何最小化？有没有替代方案？为什么仍然选择这个方案？

#### 老师想听什么

这一题是非常关键的问题。老师通常会通过这题看：

- 你们是否知道自己方案的薄弱点
- 是否考虑过替代方案
- 为什么现在的方案是合理选择

#### 回答思路

建议按“风险 - 应对 - 替代方案 - 仍然选择原因”四段回答。

#### 推荐回答

当前项目的主要风险有四类。

第一类是环境与部署风险。因为系统依赖 MySQL、Redis、Docker 和后端服务协同运行，如果某个组件未正确启动，就会影响整体链路。对此我们的控制方法是把依赖关系写进运行文档，并把判题链路拆分成 Web 服务和独立 worker，便于定位问题。

第二类是判题安全与稳定性风险。用户提交的是任意代码，如果直接在宿主机执行，会存在明显风险。因此我们采用 Docker 沙箱执行代码，并设置 CPU、内存、进程数、输出大小、只读根文件系统等限制，降低恶意代码和异常代码对宿主环境的影响。

第三类是权限与模块复杂度风险。因为系统不仅有学生用户，还有教师和管理员，不同角色之间的权限边界必须清晰。对此我们在后端通过 JWT 和角色中间件控制访问范围，并按比赛、教学、论坛、管理等模块拆分路由和服务。

第四类是测试与回归风险。当前项目已经具备较完整原型，但自动化测试还不充分，因此后续联调和回归验证仍需要继续加强，特别是在判题结果一致性、权限边界和复杂业务流方面。

替代方案方面，我们考虑过两类：一种是把判题逻辑直接写在 Web 进程里，不单独拆 worker；另一种是直接复用外部在线判题平台。前者虽然实现更简单，但不利于扩展、隔离和异步处理；后者能减少部分工作量，但不利于把教学、比赛、班级和平台管理统一到同一系统中。

所以我们仍然选择当前方案，因为它在课程项目的复杂度范围内，比较平衡地兼顾了可实现性、扩展性和展示完整度。

#### 可以补充的一句话

> 我们当前方案不是最轻量的，但它更适合支撑“判题 + 比赛 + 教学 + 管理”的统一平台目标。

### Q7. 项目的需求是什么？你们是如何识别这些需求的？

#### 老师想听什么

这一题本质上是在问你们是否做了需求工程，而不是看到别人有的功能就机械照搬。

#### 回答思路

回答时分成两部分：

- 需求有哪些
- 这些需求是怎么识别出来的

#### 推荐回答

当前项目的需求可以概括为七类：用户与权限管理、题库与题目详情、题解与内容展示、提交与判题、比赛与榜单、教学管理、社区与平台治理。也就是说，我们识别的不只是“做题”需求，还包括比赛组织、班级作业和讨论交流这些实际场景中的支撑需求。

在需求识别方法上，我们主要采用了“角色驱动 + 场景驱动”的方法。先从学生练习、教师布置作业、管理员维护平台这些典型使用场景出发，再抽象出游客、学生、教师、管理员几类参与者，最后把这些场景拆成可实现的用例、接口和页面。

#### 可以补充的一句话

> 我们的需求不是从“技术上还能再加什么”出发，而是从“不同角色在真实场景中需要完成什么任务”出发识别出来的。

### Q8. 需求来自哪些利益相关者（用户、客户、委托方）？

#### 老师想听什么

老师是在确认你们是否识别了“谁对这个系统真正有需求”。

#### 回答思路

把利益相关者分成直接使用者和间接约束者两层会比较清楚。

#### 推荐回答

从直接使用者角度看，利益相关者主要包括游客、学生、教师和管理员。游客有公开浏览需求，学生有做题、比赛、作业和讨论需求，教师有题单、班级、作业和题解管理需求，管理员有题库、比赛、公告、论坛和用户治理需求。

从间接需求来源看，课程教学场景本身和指导老师也属于重要的利益相关者，因为这个项目不仅要解决学生和教师在平台中的使用问题，还要满足课程项目对可演示性、可维护性和可扩展性的要求。

#### 简化总结

> 这个项目既服务终端使用者，也服务教学组织者和课程项目本身。

### Q9. 是否为每种类型的利益相关者绘制了场景和用例图？

#### 老师想听什么

老师不是一定要求你们每个角色单独画很多图，而是想知道你们有没有做结构化建模。

#### 回答思路

这里要实事求是。当前仓库里已经有总体用例图和多个分域用例图，因此可以明确回答“做了”，但要准确说明形式。

#### 推荐回答

我们已经围绕不同利益相关者和业务场景完成了用例建模，并绘制了总体用例图以及按业务域拆分的局部用例图。总体图覆盖游客、学生、教师、管理员四类参与者；局部图则分别细化了公开访问、判题训练、比赛、教学、社区与管理等场景。

如果严格区分“场景图”和“用例图”，目前我们更完整的是“场景分析 + 用例图”的组合：也就是先做角色与场景分析，再把核心行为整理成用例图，而不是给每个角色都再单独画一整套冗长图纸。这样更适合课程项目展示，也更容易和实际代码结构对应。

#### 可以补充的一句话

> 我们建模的目标不是把图画得越多越好，而是保证图能够真正服务后续设计和实现。

### Q10. 项目中使用了需求建模图吗？

#### 老师想听什么

老师要确认你们不是“边写代码边临时想功能”，而是做过结构化分析。

直接回答“使用了”，然后列出建模图的类型和作用。

因此可以说，我们的需求建模不是停留在文档层面的形式工作，而是和当前实现直接对应的。

可以提到当前仓库中已有：

- 总体用例图
- 教学模块用例图
- 比赛模块用例图

> 从当前项目来看，我们已经把 SEU OJ 定位成一个面向练习、比赛、教学和讨论的综合平台，而不是单一刷题网站。需求来源主要是学生、教师、管理员和课程场景本身，我们通过角色分析和场景分析识别需求，并用用例图做了结构化建模。在技术实现上，当前已经完成了 Go 后端、MySQL、Redis、Docker 沙箱和前端页面的主链路，因此技术上是可行的。后续的重点不再是重新设计方向，而是继续做好联调、测试、优化和最终交付。

## 5. 容易加分的答辩表达

下面这些表达在答辩里通常比较容易加分，因为它们体现的是项目理解，而不是机械背答案。

### 表达 1

> 我们的设计重点不是功能堆叠，而是把“训练、判题、比赛、教学、讨论”组织成一个完整业务闭环。

### 表达 2

> 我们当前不是停留在需求设想，而是已经让需求建模、接口设计和代码实现形成了对应关系。

### 表达 3

> 我们在范围控制上比较明确，优先完成课程项目可落地、可演示、可持续扩展的核心能力，而不是一次性追求所有高级特性。

### 表达 4

> 对于风险，我们没有回避，而是明确区分了环境依赖风险、判题安全风险、权限复杂度风险和测试不足风险，并针对性做了约束和拆分。

## 6. 容易被追问的薄弱点与建议口径

为了避免答辩时被动，建议提前准备以下几个薄弱点的统一口径。

### 薄弱点 1：自动化测试还不充分

建议口径：

> 当前原型系统的重点首先是打通业务链路，因此自动化测试还在补强中。后续会优先完善关键服务层和权限边界的测试，并加强联调回归。

### 薄弱点 2：部署链路依赖较多

建议口径：

> 是的，当前运行依赖 MySQL、Redis、Docker 和后端服务，这也是为什么我们把运行文档写得比较详细，并将判题 worker 独立出来，方便定位和隔离问题。

### 薄弱点 3：为什么不直接用现有 OJ

建议口径：

> 如果只做刷题，直接使用现有平台确实更省事；但我们的目标是把教学、班级、作业、题解、论坛和比赛统一起来，所以需要一个可控、可扩展、可定制的系统。

## 8. 最终建议

答辩时不要把自己逼到“必须把所有细节一口气讲完”的状态。更稳妥的方法是：

1. 每题先给结论
2. 再给一到两个最强依据
3. 最后补一句边界或下一步计划

如果现场时间紧，优先把以下四题答稳：

- Q3 技术可行性与范围
- Q5 风险与替代方案
- Q7 需求与需求识别
- Q10 需求建模图

因为这四题最能体现你们是否真的理解项目。
