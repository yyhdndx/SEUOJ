# Frontend Structure

当前前端已经从“单个超大 app.js + 单个超大 styles.css”拆分为按业务域组织的结构，目的是降低多人协作冲突，并让不同成员能在相对独立的文件内开发。

## 当前结构

- `index.html`
  - 页面壳与导航
  - 脚本与样式装配顺序

- `styles.css`
  - 前端样式入口文件
  - 仅负责通过 `@import` 引入各业务域 CSS

- `css/base.css`
  - 全局变量与基础排版
  - 顶部导航与页面壳
  - 通用按钮、表单、表格、卡片、标签、布局容器

- `css/problem.css`
  - 题库页与题目详情工作台
  - 左右分栏、代码编辑区、运行结果区
  - Markdown 题面与样例块样式

- `css/submission.css`
  - 提交详情 verdict 面板
  - 统计卡、通知块、结果摘要

- `css/contest.css`
  - 比赛榜单单元格
  - 比赛公告操作区
  - 比赛页局部状态样式

- `css/teaching.css`
  - 题单、班级、作业/考试页面
  - 教学摘要卡、进度块、题解管理区

- `css/forum.css`
  - 论坛列表、帖子详情、回复区
  - 论坛筛选、摘要卡、管理区

- `app.js`
  - 全局状态
  - 路由分发
  - 通用工具函数
  - 仍未拆走的核心函数

- `js/bootstrap.js`
  - 应用启动入口
  - 浏览器事件绑定
  - 首次渲染触发

- `js/general-pages.js`
  - 个人中心
  - 排行榜
  - 公告与公告详情
  - 公共统计表渲染

- `js/problems.js`
  - 题库列表
  - 题目详情
  - Markdown 题面展示
  - 样例运行结果区
  - 题解展示

- `js/submissions.js`
  - 我的提交列表
  - 提交详情
  - 提交轮询
  - 相关辅助函数

- `js/contests.js`
  - 比赛列表
  - 比赛详情
  - 比赛题目页
  - 榜单与比赛公告
  - 管理员比赛页面

- `js/teaching.js`
  - 题单
  - 班级
  - 作业/考试
  - 教师端页面
  - 题解管理页

- `js/admin.js`
  - 管理员题目页面
  - 管理员提交管理
  - 管理员用户管理
  - 管理员公告管理

- `js/forum.js`
  - 论坛列表
  - 论坛主题详情
  - 发帖 / 回帖 / 论坛管理交互

## 为什么这样拆

原来的主要问题：
- 所有页面逻辑都集中在一个超大文件里
- 所有样式也集中在一个超大 CSS 里
- 多人同时开发极易出现冲突
- 查找问题和定位责任模块很困难
- 一个模块改动很容易影响其他页面

拆分后的收益：
- 不同业务域有独立 JS 文件，职责更清晰
- 不同业务域也有独立 CSS 文件，样式修改范围更可控
- 组员可以按模块分工，减少 Git 冲突
- 页面问题更容易定位到对应域文件
- 后续继续重构时，可以逐步把 `app.js` 收缩成真正的核心壳，把 `base.css` 收缩成真正的全局样式层

- `js/forum.js`

### 公共维护者
- `app.js`
- `js/bootstrap.js`
- `js/general-pages.js`
- `index.html`
- `styles.css`
- `css/base.css`

## 样式加载方式

当前样式不是在 `index.html` 里逐个 `<link>` 引入，而是：

1. `index.html` 只加载 `styles.css`
2. `styles.css` 再通过 `@import` 引入分域 CSS

这样做的好处是：
- 页面入口不需要频繁改动
- 后续继续拆分 CSS 时，改动集中在 `styles.css`
- 对现有静态前端结构侵入更小

## 继续优化建议

如果后续还要进一步协作优化，建议继续做：

1. 把共享渲染函数继续抽离
- Markdown 渲染
- 状态标签
- 表格与卡片渲染
- 时间格式化和状态格式化

2. 给每个域模块补最小 README
- 说明文件负责哪些页面
- 哪些函数是公共依赖
- 哪些路由由该模块负责

3. 给样式命名做进一步约束
- 统一前缀
- 明确通用类与域类边界
- 减少不同模块间类名碰撞

## 当前结论

前端已经从“单文件巨石”演进为“按业务域拆分的 JS + CSS 双层结构”，这已经足够支持小组内分模块协作开发。后续组员可以在不频繁互相冲突的前提下，分别完善比赛、教学、论坛、管理后台、题目工作台等页面。
